# Council Vote: Chiral Validation — The Constraint as Feature

## Vote ID: 18aab433f2a33997

## Purpose
To formalize chiral validation as a design constraint for Stoneclad, ensuring that no substrate validates itself. This approach aims to enhance security by requiring cross-substrate confirmation for critical decisions.

## Background
During the MOCHA sprint, it was identified that Stoneclad's current validation mechanisms rely on the same infrastructure and models, creating a potential security vulnerability. To address this, the concept of chiral validation was proposed, where different substrates (e.g., different nodes or models) are used to validate each other.

## Proposal
### Key Points
- **Security Self-Audit**: Security audits must run on a different node or model than the system being audited.
- **Carbon Validates Silicon**: Partner (human) reviews security-critical council decisions.
- **Silicon Validates Carbon**: Stoneclad surfaces anomalies in 96K memories that Partner cannot manually audit.
- **Formal Design Constraint**: Chiral validation will be formalized as a design constraint, ensuring that all security-critical decisions require cross-substrate confirmation.

### Implementation
- **Node Assignment**: Ensure that security audits and critical decision-making processes are distributed across different nodes.
- **Model Diversity**: Use multiple models for validation to reduce the risk of shared vulnerabilities.
- **Audit Logs**: Maintain detailed logs of cross-substrate validation processes for review and auditing.

## Voting Options
- **Option A**: Approve the chiral validation design constraint.
- **Option B**: Reject the chiral validation design constraint.
