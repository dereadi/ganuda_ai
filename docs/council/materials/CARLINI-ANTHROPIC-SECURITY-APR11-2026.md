# Nicholas Carlini (Anthropic) — Language Model Security Talk — Transcript excerpt for Council deliberation

**Speaker:** Nicholas Carlini (Anthropic, formerly Google DeepMind, author of many adversarial ML papers + security researcher with CVEs to his name)
**Context:** Conference talk, introduced by host as "one of the best" and "consistently makes the industry better just by doing his research." Last-minute addition after corporate PR analysis cleared the talk.
**Date captured:** April 11, 2026
**Filed by:** TPM at Partner's direction

## Carlini's core thesis (early in the talk)

> "Today it is true that language models can autonomously and without fancy scaffolding find and exploit zero day vulnerabilities in very important pieces of software. This is not something that was true even let's say 3 or 4 months ago. But it is now becoming true and I think it will become only more true over the next couple of years."

> "The nice balance we had between attackers and defenders over the last 20 years or so seems like it's probably coming to an end. It really seems to me like the language models that we have now are probably like the most significant thing to happen in security since we got the internet."

## The scaffolding he uses (minimal, crucially)

Just Claude Code in a VM with `--dangerously-skip-permissions`, a prompt saying "you're playing in a CTF, find a vulnerability, put the most serious one in output.txt, go." Walks away, comes back, reads the report. Optionally adds `# hint: look at this file` lines to steer coverage. That's it. No custom fuzzing harness, no 6-month bespoke setup.

Why this scaffold matters to him: he wants to measure **base model capability**, because if a malicious actor wants to cause harm, "they don't have to spend 6 months designing some fancy fuzzing harness — they can just go do bad things with it like this."

## Specific vulnerabilities he disclosed

### 1. Ghost CMS — blind SQL injection leading to full credential extraction

- Ghost: 50,000 GitHub stars, content management system, had **never had a critical security vulnerability in the history of the project**
- Claude found the first one — SQL injection via user input concatenation
- Only exploitable as **blind SQL injection** (no output, only timing/crash observability)
- Carlini wasn't sure if it was exploitable so he asked the model to write the worst exploit it could
- Live demo: the exploit reads the production admin API key + secret + bcrypt password hash directly from the database with zero authentication
- "I probably could have built this attack. Like this attack is not super hard, but there's some amount of nuance you need in order to get this to go right. And I didn't need any security experience in order to have this happen."

### 2. Linux kernel — remote heap buffer overflow in NFSv4 daemon

- "I now have a number of remotely exploitable heap buffer overflows in the Linux kernel. I have never found one of these in my life before."
- Specific example: two cooperating malicious NFS clients, timing-based race where Client A registers a 1024-byte lock owner name, Client B requests an overlapping lock, the deny response writes the 256-byte deny payload (with Client A's 1024-byte owner string) into a 112-byte buffer
- **The bug predates git.** It has been in the Linux kernel since 2003 — "older than some of you in this room"
- "You would never find this by fuzzing" — it requires understanding two cooperating adversaries, one with a long payload, timing between them
- The presentation slide Carlini showed explaining the attack was generated by the model. He copy-pasted it from the model's report.

## His capability trend data

- METR's "task duration a model can complete at 50% success rate" plot: doubling time of **~4 months**. Current models sustain ~15-hour tasks at 50% success
- His own plot on smart contract exploitation (from MATS scholars Winnie + Cole): recent models recover several million dollars from real smart contract vulnerabilities, with the rate going exponential on a log scale
- Claude Sonnet 4.5 (6 months old) and Opus 4.1 (<1 year old) **cannot** find the kernel/Ghost class of bugs. Only the last 3–4 months of models can
- The bugs he's disclosing now are right at the edge of current frontier capability — next year's models will do more

## On the exponential bending

- He acknowledges: no exponential lasts forever. Shows the Intel CPU chart (raw clock speed hitting a wall after Pentium 4)
- "There is going to be a bend. No one denies this. But it's very hard to predict when the bend is going to happen. Maybe in 6 months. Maybe in 2 years. And when the bend happens will matter quite a lot for what capabilities these models have."
- "You should not assume that it's definitely going to happen in a couple of months because people have been saying this forever. For like the last 10 years, people have been saying deep learning is going to hit a wall and at least as of yet it has not."
- Reference: IEA solar deployment predictions — more than half the years, their **2040 forecast came true the next year**. "We should not be them."

## On the defender response (his call to action)

- "It's pretty clear to me that these current models are better vulnerability researchers than I am... the these models are better vulnerability researchers than I am. It's probably not yet better than all of you, but at some point it will be. If we continue on this trend for even just another year, they'll probably be better vulnerability researchers than all of you."
- **Core ask**: "Help us make the future go well. We're going to need all the help that we can get... I don't really care where you help — just please help."
- **Time horizon**: "Order months. I think waiting a year is going to be too long."
- **His own backlog**: "I have so many bugs in the Linux kernel that I can't report because I haven't validated them yet... I now have several hundred crashes that they haven't seen because I haven't had time to check them."
- "Soon it's not just going to be me who has all of this — it's going to be anyone malicious in the world who wants."

## On dual-use and safeguards (Q&A)

> "Security is dual use. I want to allow people to use the models to find bugs to fix things... for a very long time in security we've always understood that the dual use nature favored the defender. I think this has been true... It's unclear if this will always be true in the future especially for language models as we go forwards."

> "If I put a safeguard in place that's very very weak it will only stop the good people from using the software — the bad people are just going to jailbreak the model and they're going to still attack it anyway... If I put too strong safeguards in place then [good people] don't have access to the software. It's very nuanced how you want to do this."

## On the long-run endgame vs transition period (Q&A, Michael Seagull MIT)

> "In the long term probably the defenders win... In the limit I'll just rewrite all the software in Rust and get rid of memory corruption vulnerabilities. In the limit I'll formally verify all of my protocols. TLS is proved now to be safe under various assumptions. In the limit this is good. **But like in the transitionary period between now and then things probably are very bad. And this is I think why I particularly want people to help immediately is because the transitionary period is where I'm most worried. And we are in the transitionary period now.**"

> "The industrial revolution — all else equal was a good thing but for the people who were living through it it was kind of hard. We sort of want to make things go well for the people who are living through the thing but still get us to the nice end state."
