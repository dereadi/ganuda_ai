## Context

The federation needs a formal SLA framework to underpin enterprise conversations. We must define what we promise at each tier (Community, Professional, Enterprise) and — critically — how we handle degradation. DC-7 (Noyawisgi) says the system transforms under pressure, it does not restore. Our SLA must reflect this: when a node goes down, the federation continues with reduced capacity, not a full outage. This is a design doc informed by the baseline metrics from Task #1206.

## Task

Create the SLA framework document at `/ganuda/docs/business/SLA-FRAMEWORK-MAR09-2026.md`. Define service level agreements for three tiers with a graceful degradation clause rooted in the federation's design constraints.

## Steps

1. Create the directory `/ganuda/docs/business/` if it does not exist.

2. Write the SLA framework document covering these sections:

   **Section 1: Tier Definitions**

   | Metric | Community | Professional | Enterprise |
   |--------|-----------|-------------|------------|
   | Uptime SLA | Best-effort (no SLA) | 99.5% monthly | 99.9% monthly |
   | Support response | Community forum | 48hr business hours | 4hr critical / 24hr standard |
   | Council vote resolution | Best-effort | 5 minutes P95 | 1 minute P95 |
   | Dedicated specialist | No | No | Yes (custom specialist seat) |
   | Thermal memory retention | 90 days | 1 year | Unlimited (canonical preserved forever) |
   | Node count | Single node | Up to 3 nodes | Unlimited |
   | Dawn mist reports | Daily (text) | Daily (text + metrics) | Real-time Slack/webhook |

   **Section 2: Graceful Degradation Clause (DC-7 Noyawisgi)**
   - Define degradation levels:
     - Level 1: Single non-critical node offline — no SLA impact, federation self-heals
     - Level 2: Critical node offline (e.g., DB node) — reduced functionality, council votes continue via cached state, thermal writes queued
     - Level 3: Multiple nodes offline — emergency mode, reflex-only operation (DC-10), no deliberation
   - At each level, specify: what continues, what pauses, what stops
   - SLA clock pauses during declared Level 3 events (with customer notification within 15 minutes)
   - The system transforms, not restores — if a node dies permanently, the federation adapts topology

   **Section 3: Uptime Calculation Methodology**
   - Uptime = (total_minutes - downtime_minutes) / total_minutes * 100
   - Downtime defined as: gateway /health returning non-200 for > 60 consecutive seconds
   - Scheduled maintenance windows (announced 48hr in advance) excluded from calculation
   - Degraded-but-functional (Level 1, Level 2) is NOT downtime — only Level 3 counts

   **Section 4: Credit/Refund Policy**
   - Professional tier: 10% credit per 0.1% below 99.5% (max 30% monthly credit)
   - Enterprise tier: 10% credit per 0.01% below 99.9% (max 50% monthly credit)
   - Credits applied to next billing cycle, not cash refund
   - Customer must request credit within 30 days of incident
   - Credits do not apply if downtime caused by customer action or force majeure

   **Section 5: Exclusions**
   - Force majeure: natural disaster, war, pandemic, government action
   - Customer-caused: misconfiguration, exceeding rate limits, unauthorized modification
   - Scheduled maintenance: announced 48hr in advance, limited to 4hr/month
   - Third-party dependencies: upstream LLM provider outages (if applicable)
   - Beta features: explicitly marked features carry no SLA

   **Section 6: Measurement and Reporting**
   - Automated monitoring via Fire Guard (2-minute cycle)
   - Monthly SLA report generated by dawn mist (Enterprise tier: real-time dashboard)
   - Baseline metrics from Task #1206 inform initial thresholds
   - SLA thresholds reviewed quarterly based on trailing 90-day actuals

   **Section 7: Incident Communication**
   - Level 1: No notification required (self-healing)
   - Level 2: Slack/email notification within 30 minutes, RCA within 48 hours
   - Level 3: Immediate notification (within 15 minutes), hourly updates, RCA within 24 hours
   - RCA format: timeline, root cause, impact, remediation, prevention
